## Overview

API SICARX is a FastAPI middleware between an e-commerce frontend (Ferretería Charly) and the SICAR X POS/ERP platform. It exposes a small REST API, secured with a static API key, that orchestrates calls to SICAR X's REST and GraphQL endpoints while maintaining a local PostgreSQL cache of the product catalog for fast reads.

## Commands

- Run dev server: `uvicorn app.main:app --reload`
- Run the catalog sync worker (separate long-running process, not started by the FastAPI app): `python -m app.worker.sync_task`
- Migrations: `alembic revision --autogenerate -m "description"`, `alembic upgrade head`, `alembic downgrade -1`
- Install deps: `pip install -r requirements.txt`
- No test suite exists in this repo currently.

### Docker Compose

`docker-compose.yml` runs the whole stack: `db` (Postgres 16), a one-off `migrate` service (`alembic upgrade head`, runs once and exits), `api` (uvicorn), and `worker` (`sync_task`) — `api`/`worker` both wait on `migrate` completing successfully before starting, so migrations never race.

- `docker compose up --build` — build images and start everything
- `docker compose logs -f api` / `worker` — tail a service's stdout (app-level logs still also land in `app.log`/`sync.log` on the host via bind mounts)
- `docker compose run --rm migrate alembic revision --autogenerate -m "description"` — generate a new migration without starting the rest of the stack
- `docker compose down` — stop everything; add `-v` to also drop the `db_data` volume (destructive — wipes the containerized Postgres)

**Important**: the `db` service creates a **new, empty** Postgres instance (its own `db_data` volume) — it is not connected to whatever Postgres you already use for local development or production. `DATABASE_URL` for `api`/`worker`/`migrate` is overridden in `docker-compose.yml` to point at this containerized `db` (not the `DATABASE_URL` in your `.env`, which is only used for the other vars). To point this stack at your existing Postgres instead, remove the `db` service and its `depends_on` entries, and set `DATABASE_URL` in `.env` to that instance's address (use `host.docker.internal` in place of `localhost` if it runs on the host machine, since `localhost` inside a container refers to the container itself).

### Deploying to Railway

Live at project **api-sicarx** (workspace: Angel Villalvazo's Projects), three services: `api` (public domain `api-production-cf7a.up.railway.app`), `worker` (no public networking), and a Postgres plugin named **`Postgres-O4xA`** (mind the suffix — an earlier duplicate `Postgres` plugin from a setup mistake has been deleted).

Two Railway services run from this same repo/`Dockerfile`: `railway.api.json` (uvicorn, plus `alembic upgrade head` as a `preDeployCommand`) and `railway.worker.json` (`sync_task`, no pre-deploy command — see below on why only one service runs migrations). Railway's managed Postgres plugin replaces the `db`/`migrate` services from Docker Compose — there is no dedicated one-off migration service on Railway, so `preDeployCommand` on the `api` service is what runs migrations instead.

Both services are now connected to the `AMiguelV05/API-SICARX` GitHub repo (`main` branch) via `railway service source connect`, for auto-deploy-on-push. Each service's Settings → *Config-as-code file path* is set (`api` → `railway.api.json`, `worker` → `railway.worker.json`) — confirmed working: a later GitHub-triggered deploy picked up both files correctly (`configFile: /railway.api.json` / `/railway.worker.json` in the deployment record).

Without that per-service path set, Railway can't find a matching `railway.json`/`railway.toml` at the repo root (there isn't one committed — see below) and silently falls back to the bare `Dockerfile`'s default `CMD`. **This was confirmed to actually break both services** the moment GitHub was first connected, before the path was set: `api` crash-looped on the hardcoded port 8000 instead of Railway's `$PORT` with no migration ever running, and `worker` came up running `uvicorn` (the API's own default command) instead of `sync_task` — i.e. two copies of the API server, no sync worker at all. Fixed by a manual `railway up` redeploy with the correct config, then permanently by setting the Config-as-code path per service.

There's deliberately no root `railway.json` committed — with only one root config file, both services would resolve to whichever one happens to be there (see the incident above). Manual deploys instead stage the right file temporarily: `cp railway.api.json railway.json && railway up --service api`, then `cp railway.worker.json railway.json && railway up --service worker`, deleting the root file afterward each time.

**Auto-deploy-on-push has not been reliable in practice** — a real push to `main` (the search-endpoint commit) sat for a while without triggering a build on either service; nothing in `railway deployment list`/`railway status` showed it queued or building. Per [Railway's GitHub auto-deploy docs](https://docs.railway.com/deployments/github-autodeploys.md), this symptom (branch/watch-paths correct, commit genuinely pushed, still nothing triggers) points at a GitHub App permission/access issue — a pending permission approval on the app installation, or the connected account lacking contributor access — neither of which is inspectable from the Railway CLI. Applied the docs' own recommended fix: `railway service source disconnect --service <name>` then `railway service source connect --repo AMiguelV05/API-SICARX --branch main --service <name>` for **both** `api` and `worker`. Reconnecting itself immediately triggered a fresh build on the current latest commit for both services (confirmed via `commitHash` in the deployment record matching `git log -1`), and both came back healthy.

If a push doesn't seem to be deploying, check `railway deployment list --service <name> --json` for a recent entry with `"reason": "deploy"` and a matching `commitHash` before assuming it worked — don't rely on the push alone. Immediate workaround for any single stuck deploy: `railway redeploy --service <name> --from-source --yes`, which pulls and deploys the latest commit on the connected branch right away, without needing a new push or webhook. The disconnect/reconnect cycle above is the fix for the underlying trigger itself; **still unverified whether it fixes the recurrence** — the next real push is the real test, since a manually-triggered reconnect deploy doesn't prove the webhook itself is now healthy. If the next push doesn't auto-deploy either, check GitHub's installed-app settings (Settings → Integrations → GitHub, or the `AMiguelV05` org's installed-apps page) for a pending permission banner before troubleshooting further on the Railway side.

**Gotchas specific to Railway** (neither applies to the Docker Compose setup):
- Railway does **not** run `startCommand` through a shell — `--port $PORT` gets passed to uvicorn as the literal 5-character string `$PORT`, not the actual port number, and it crash-loops with `Error: Invalid value for '--port': '$PORT' is not a valid integer.` (hit and fixed live during initial deploy). `railway.api.json`'s `startCommand` must wrap the whole thing in a shell explicitly: `sh -c \"uvicorn app.main:app --host 0.0.0.0 --port $PORT\"`. The worker's command has no env var references so it doesn't need this.
- Railway's Postgres plugin exposes `PGHOST`/`PGPORT`/`PGUSER`/`PGPASSWORD`/`PGDATABASE` (and a `DATABASE_URL` using the plain `postgresql://` scheme) — but `app/core/database.py` needs the `asyncpg` driver. Don't use the plugin's own `DATABASE_URL` as-is; set `DATABASE_URL` on both `api` and `worker` manually as a variable reference: `postgresql+asyncpg://${{Postgres-O4xA.PGUSER}}:${{Postgres-O4xA.PGPASSWORD}}@${{Postgres-O4xA.PGHOST}}:${{Postgres-O4xA.PGPORT}}/${{Postgres-O4xA.PGDATABASE}}`.
- Both services import `app.core.config.settings`, and pydantic-settings fails at import time if any required var is missing — so all 7 vars (`X_API_KEY`, `SICAR_ADMIN_EMAIL`, `SICAR_ADMIN_PASSWORD`, `SICAR_TOKEN`, `SICAR_PRICE_LIST_ID`, `CASH_REGISTER_UUID`, plus the `DATABASE_URL` override above) must be set on **both** `api` and `worker`, even the ones only one of them actually uses.
- Only `api` runs `alembic upgrade head` (as its `preDeployCommand`) — Railway has no `depends_on`/completion-order primitive across services like Compose's `service_completed_successfully`, so avoid a migration race by not running it from both. On the very first deploy, deploy `api` first and let it finish before deploying `worker`, so the schema exists before the worker's first sync tries to write to it (this was the actual order used). On later deploys that include a new migration, redeploy `api` before `worker` for the same reason.
- The worker logs to `sync.log` via `RotatingFileHandler` inside the container's ephemeral filesystem, same as locally — Railway has no bind-mount equivalent, so `railway logs --service worker` only ever shows `Starting Container` and nothing else. To confirm the worker is actually syncing, check `POST /catalog`'s `total` count over time instead (both services share the same Postgres).

### Incident: Postgres-O4xA crashed from a full volume (2026-07-13)

`Postgres-O4xA`'s volume is only 500MB (Railway's Postgres plugin default). During a full catalog sync (~124k products, upserted via `sync_task.py`'s paginated `on_conflict_do_update`), WAL grew faster than checkpoints could reclaim it, and a restart triggered crash recovery that needed more temp WAL space than was left — Postgres got stuck in a redo loop, failing every attempt with `FATAL: could not write to file "pg_wal/xlogtemp.*": No space left on device`. `railway logs --service Postgres-O4xA` shows this directly.

Since this Postgres holds only a disposable cache of SICAR X's catalog (SICAR X is the real source of truth; the worker fully re-syncs it on its own), recovery was to **delete the corrupted volume and let it reinitialize empty** rather than try to repair the WAL in place — confirmed with the user first, since it's irreversible. Sequence: `railway volume delete --volume <id> --yes`, wait for Railway to actually process the deletion (`isPendingDeletion` stayed `true` for several minutes — this is async and not forceable from the CLI), then the volume came back healthy on its own once that finished. From there: `railway redeploy --service api` (re-runs `alembic upgrade head` on the now-empty database) → `railway redeploy --service worker` / `railway restart --service worker` to force an immediate resync rather than waiting for the next 5-minute interval.

**Still open**: the volume is still only 500MB — the same condition that caused this crash. If another full sync (or one that runs alongside natural production writes) generates enough WAL again, this can recur. Consider bumping the volume size in the dashboard (CLI has no resize command, only `add`/`delete`/`update` for name/mount-path) to something like 1–2GB for headroom, or tuning `max_wal_size`/checkpoint frequency, before the next time this much data churns through in one go.

**It recurred the same day, unprompted** — a few hours later, `Postgres-O4xA` re-initialized from a completely empty data directory again (`PostgreSQL init process complete; ready for start up` in `railway logs --service Postgres-O4xA`, i.e. a fresh init, not just a container restart), with no volume-delete command run this time. Cause unconfirmed — possibly the earlier volume-deletion operations were still settling in the background and only actually took effect at that point, possibly something else on Railway's side. Same recovery worked again: `railway redeploy --service api --from-source` (re-ran all migrations, including the newer `224799e4444b` trigram one, cleanly on the empty DB) → `railway restart --service worker` to force an immediate resync. Confirmed recovered both times by watching `POST /catalog`'s `total` climb back up. Given this has now happened twice in one day, treat the 500MB-volume gotcha above as more urgent than "eventually" — and if `/catalog` or `/search` ever returns suspiciously low/zero counts, check `railway logs --service Postgres-O4xA` for a fresh-init line before assuming it's an application bug.

## Configuration

- Settings load via pydantic-settings from a project-root `.env` (`app/core/config.py`, `Settings`). Required vars: `DATABASE_URL`, `X_API_KEY`, `SICAR_ADMIN_EMAIL`, `SICAR_ADMIN_PASSWORD`, `SICAR_TOKEN`, `SICAR_PRICE_LIST_ID`, `CASH_REGISTER_UUID`, `CLIENT_JWT_SECRET` (optional: `CLIENT_JWT_EXPIRE_MINUTES`, defaults to 10080 / 7 days).
- `DATABASE_URL` must use an async driver (`postgresql+asyncpg://...`) — `app/core/database.py` uses `create_async_engine`.
- `secret.py` is a standalone one-off script (`secrets.token_urlsafe(60)`) for generating a new `X_API_KEY`; it isn't imported by the app. `CLIENT_JWT_SECRET` needs the same kind of random value (e.g. `secrets.token_urlsafe(48)`) — it signs client-account JWTs, so it must stay a real secret, never a hardcoded default.
- **New required var alert**: `CLIENT_JWT_SECRET` was added for the client-accounts feature below. Per the Railway gotchas further down, pydantic-settings fails at import time if it's missing — this must be set on **both** `api` and `worker` on Railway before the next deploy of either, same as every other required var, even though only `api` actually issues/needs it.
- **New required var alert**: `MP_ACCESS_TOKEN`, `MP_WEBHOOK_SECRET`, `FRONTEND_BASE_URL`, and `API_BASE_URL` were added for the Mercado Pago payment integration below (`app/services/payment_service.py`). Same deal as `CLIENT_JWT_SECRET` — must be set on **both** `api` and `worker` on Railway even though only `api` uses them, or pydantic-settings fails at import for both services. `MP_ACCESS_TOKEN` is the private/secret token from the Mercado Pago dashboard (never the public key — that one belongs in the frontend's own env, not here). `FRONTEND_BASE_URL` is the frontend's real domain (used for the preference's `back_urls` — Mercado Pago rejects `localhost`/`127.0.0.1` there). `API_BASE_URL` is this API's own public domain (`https://api-production-cf7a.up.railway.app` in production) — used to build `notification_url` so Mercado Pago's webhook can reach `POST /v1/payments/webhook`.
- `app/core/database.py`'s async engine sets `pool_pre_ping=True` (discards stale connections instead of surfacing a confusing error — relevant given Postgres-O4xA has restarted unexpectedly more than once, see the incident below) and explicit `pool_size=5`/`max_overflow=5` — deliberately conservative since `api` and `worker` are two independent processes, each with their own pool, against a Postgres plan that's already been resource-constrained once.

## Architecture

Two separate processes share the same Postgres database and `Product` model:

1. **FastAPI app** (`app/main.py`) — serves the HTTP API consumed by the e-commerce frontend.
2. **Background worker** (`app/worker/sync_task.py`) — a standalone script with its own logging setup and its own `AsyncIOScheduler` that polls SICAR X every 5 minutes and upserts the full catalog into `products`. Must be run as a second process alongside uvicorn.

### Two distinct SICAR X auth schemes — do not conflate them

- **Admin/B2B token** (`app/services/sicar_auth.py`, the `sicar_auth` singleton): one shared token for server-to-server calls (catalog sync, product-detail scraping, payment, cancellation). Seeded from `SICAR_TOKEN`, held in memory, refreshed via `SicarAuthManager.refresh_token()` (hits two AWS Lambda endpoints with `SICAR_ADMIN_EMAIL`/`SICAR_ADMIN_PASSWORD`) whenever SICAR X returns 401. Every caller follows the same pattern: try with the current token, on 401 call `refresh_token()`, retry once (see `order_service.pay_order_in_sicar`, `cancel_service.process_order_cancellation`, `product_service.fetch_full_details_from_sicar`).
- **Customer/session token** (`app/services/session_service.py`): a per-shopper JWT. A new session is bootstrapped by scraping the `tmpStore` cookie off the SICAR X storefront HTML — SICAR double-encodes it (URL-encode, then Base64 over that, and the Base64 payload itself is URL-encoded JSON), so decoding requires URL-decode → Base64-decode → URL-decode, in that order, before `json.loads`; getting this order wrong throws `binascii.Error: Incorrect padding`. An existing session is refreshed by calling SICAR X's `/api/ecommerce/config` with the current token. This token travels in the `Authorization` header between the frontend and this API and is what's used for cart validation and order creation — never the admin token.

### Request flow: placing an order (`app/api/routes/orders.py`)

The `/orders` request body is intentionally minimal — the frontend sends only `products:
[{uuid, quantity}]` and `deliveryInfo`; everything else (pricing, tax fields, sku,
description, unit, totals) is computed server-side. This contract and the exact field
semantics below were reverse-engineered by capturing one real accepted order from the
actual storefront checkout — see `app/services/order_service.py:build_order_payload`.

1. Client sends its session JWT via `Authorization` header → `get_or_refresh_customer_session` validates/refreshes it (and now also decodes the JWT's `jti` claim as a fallback `contentId`).
2. `validate_cart_items` (`order_service.py`) checks stock/availability in SICAR X via GraphQL using the customer token, and returns the raw per-product `priceList`/`type` data plus `content.units` (sales-unit uuid → short name) for the next step — it's no longer validate-and-discard.
3. `build_order_payload` (pure function, no network calls) assembles the full SICAR order document: `sku`/`description` come from the local `Product` cache, `priceBaseTax`/`priceTax` use `priceList.netPrice1` (the tax-inclusive retail unit price) despite the field names — SICAR X does not want the tax broken out here. `amountTax` is **not** the same value — it's the line **total** (`netPrice1 * quantity`), confirmed by capturing a real order from the storefront (`agent-browser` against `ferreteriacharly.sicarx.shop`) for a `quantity > 1` line. **Incident (2026-07-21)**: this codebase previously sent `amountTax = netPrice1` (the unit price) for every line regardless of quantity — invisible for `quantity == 1` (unit price and line total coincide), but any cart with a `quantity > 1` line got a `409` from `POST /api/cart/order` with `{"code": 0, "message": "precio alterado|<uuid>"}` ("price altered"), silently blocking checkout for bulk quantities. Fixed by computing `amountTax` from `netPrice1 * quantity` per line instead of reusing the unit price. `serie` is hardcoded `"TL"` and there's no shipping line item for either delivery type — confirmed from the captured real requests. `deliveryType` is a Pydantic `Literal["PICKUP", "DELIVERYMAN"]` so an unsupported value is a clean `422` instead of reaching SICAR X and failing there.

**Home delivery (`DELIVERYMAN`) is implemented**, on top of the real payload shape captured live (`agent-browser` against the storefront, same session as the pricing-bug investigation above) by placing a real order with "Enviar a domicilio" selected instead of "Recoger en tienda". The frontend does **not** send an inline address — `deliveryInfo.addressUuid` references one of the client's saved `ClientAddress` rows (`GET /auth/me/addresses`), resolved server-side in `routes/orders.py` via `address_service.get_owned_address` (404 if it doesn't exist or belongs to another client, same pattern as every other ownership check in this codebase) before `build_order_payload` ever runs — `build_order_payload` itself needed zero changes, it just forwards whatever `delivery_info` dict it's handed. A `model_validator` on `DeliveryInfo` (`schemas/orders.py`) makes the pairing unambiguous: `addressUuid` is required for `DELIVERYMAN` and forbidden for `PICKUP` (`422` either way if violated). The resolved `ClientAddress` is translated into the exact SICAR shape — `{street, extNumber, intNumber, district, city, county, state, zipCode, country, reference}` — with `district` sourced from `neighborhood`, `reference` from `references`, and `country` **hardcoded to `"MEX"`** regardless of what's stored in `ClientAddress.country` (this business is Mexico-only; SICAR wants the ISO3 code, not a display name like "México"). Since `ClientAddress`'s fields are all optional at the address-book level but SICAR needs a complete address to actually deliver, `routes/orders.py` checks `street`/`city`/`county`/`state`/`zipCode`/`extNumber` are all populated on the resolved address and returns a `400` naming whichever are missing, rather than sending a null-heavy address to SICAR and surfacing an opaque rejection. `ClientAddress` gained a `county` column (migration `6702e8cd3eba`) — SICAR needs both `city` and a separate `county`/municipio, which the address book didn't store before. No separate shipping-cost line item is sent to SICAR for either delivery type, and none is added to the Mercado Pago charge — `ecOrderDto.total`/`OrderResponse.amount` stay product-total-only. `Order` gained a `delivery_cost` column in the same migration as a **placeholder only** — nullable, intentionally unused, reserved for a future shipping-fee integration (envia.com) that isn't built yet.

Price/total arithmetic uses `Decimal` (converted via `Decimal(str(x))`, never `Decimal(float)` directly, to avoid inheriting binary float representation error), summed and quantized to 2 decimals with `ROUND_HALF_UP` — plain `float` summation across a multi-item cart could drift the total by a cent from what SICAR X itself computes.
4. `create_order_in_sicar` posts the order to SICAR X's REST endpoint (customer token), then decrements local `Product.stock` in Postgres.
5. `create_local_order` (`services/order_history_service.py`) persists the order to the local `orders` table as `status="TO_PAY"`, linked to the authenticated `ClientAccount` — see "Client accounts linked to orders" below. `pay_order_in_sicar` (SICAR REST, admin/B2B token, 401-retry) **no longer runs here** — it now only runs once a real Mercado Pago payment is confirmed approved, from `finalize_order_payment` (see "Payments with Mercado Pago" below). Note: this now matches what the real storefront does natively — it also leaves pickup orders in `TO_PAY` until paid; this backend used to pay immediately via the admin token as an intentional shortcut, before real online payment existed.
6. `payment_service.create_preference` creates a Mercado Pago preference for the order (non-fatal if it fails — see below) and `POST /orders` returns `preferenceId`/`amount` alongside the usual `id`/`orderUuid`, for the frontend to render the Payment Brick.

Cancellation (`app/services/cancel_service.py`) mirrors this: resolve the real document `uuid`, cancel in SICAR X with the admin token, then restore local stock. SICAR X identifies each document with **two distinct values**: the `id` this API returns from order creation (Mongo-style, e.g. `6a4fd308da77fe7cd25d1dd9` — what `OrderResponse.id`/`OrderCancel.uuid` actually carries) and a separate RFC4122 `uuid` that `POST https://api.sicarx.com/documents/v1/sale/cancel` requires — passing the `id` there fails with `"is not a valid UUID"`. `_resolve_document_uuid` bridges the two with a GraphQL lookup (`document-graph/v1/graph-v2`, query `generatedV2(objectId: "<id>") { uuid }`, admin token), confirmed against a real cancellation captured live from `app.sicarx.com`. Note separately: SICAR X's own admin panel requires a step-up "Autorización" (a second login from a higher-privilege user) before it will cancel *any* order through its UI — that's a UI-level safeguard on the restricted account used for browser testing, not a constraint on this API's admin B2B token, which already has cancel privileges.

### Local catalog vs. live SICAR X data

- `Product` (`app/models/product.py`) is the single local table. The sync worker upserts it with `on_conflict_do_update` on `sicar_uuid`. Rows are never deleted — a completed sync pass marks stale rows (whose `last_sync_id` doesn't match the current pass) `is_deleted=True` instead.
- `GET /products/{uuid}` (`routes/products.py`) serves from Postgres but lazily refreshes `description_details`, `tags`, `additional_images`, etc. from SICAR X's GraphQL API when `details_updated_at` is null or older than 24h (`fetch_full_details_from_sicar`). The lookup also filters `is_deleted == False, is_active == True` — same as `/catalog`/`/search` — so a discontinued or storefront-hidden product's detail page 404s instead of still resolving directly by uuid.
- `POST /catalog` reads only from Postgres (`catalog_service.get_local_catalog`) — no live SICAR X calls — filtered by `department_uuid`/`category_uuid`/`in_stock` (`stock > 0`)/`tag` (containment match against `Product.tags`, a `JSONB` column with a `gin (tags jsonb_path_ops)` index — `ix_products_tags_gin`, confirmed via `EXPLAIN` to use a Bitmap Index Scan) with pagination, and sortable via `sort_by` (`price_asc`/`price_desc`/`name_asc`, validated by a Pydantic `Literal` so an invalid value is a `422` rather than silently ignored). No `sort_by` means no explicit `ORDER BY` — natural DB order, not guaranteed stable across requests. `Product.price` is `Numeric(10, 2)` (not `Float`) — money needs exact decimal precision; the sync worker (`sync_task.py`) writes it via `Decimal(str(...))`, never a raw Python `float`, to avoid inheriting binary-float representation error. `Product.tags` was originally a plain `JSON` column — switched to `JSONB` (migration `069dd8677786`) specifically so the `tag` filter above could use the `@>` containment operator with a GIN index; `additional_skus`/`additional_images` stay plain `JSON` since nothing queries inside them.
- `POST /search` (`catalog_service.search_products`) does a case-insensitive substring match (`ILIKE '%q%'`) against `sku` OR `name`, accelerated by `pg_trgm` GIN indexes (`ix_products_sku_trgm`/`ix_products_name_trgm`, migration `224799e4444b`) — confirmed via `EXPLAIN` to use a Bitmap Index Scan rather than a sequential scan. Plain substring search on 124k+ rows would be far too slow without these; a fresh Postgres (e.g. after the crash-recovery incident above, or a new environment) needs this migration applied before `/search` is usable at all — `CREATE EXTENSION pg_trgm` requires superuser/extension privileges, which Railway's managed Postgres plugin grants by default but a locked-down externally-managed Postgres might not. Also accepts the same `department_uuid`/`category_uuid` filters as `/catalog`, applied as a plain equality `AND` alongside the trigram match, plus an `in_stock` boolean to restrict to `stock > 0`. Results are ranked with a `CASE WHEN`-based `ORDER BY`: rows where `sku`/`name` **starts with** `q` sort before rows that only contain it mid-string (secondary sort by `name`) — confirmed via `EXPLAIN` that this sort stays cheap (in-memory `Sort` over the already-small filtered set, index scan untouched) and via live query that the tier boundary lands exactly where the true prefix-match count says it should.
- The original plain B-tree indexes on `sku`/`name` (`ix_products_sku`/`ix_products_name`) were dropped in migration `6854715fa27b` (~17MB reclaimed) — fully superseded by the trigram GIN indexes above once `/search` existed, and confirmed via codebase search that nothing else does an exact-match query or a real DB-level `ORDER BY` on either column (the model's `Column(...)` no longer sets `index=True` for them either). Found while investigating Railway's Postgres dashboard showing several "0 scans, candidate for removal" indexes — most of those (department_uuid, the trigram ones) are still genuinely used by live code paths despite showing 0 scans (the counters reset when Postgres re-initialized during the crash incidents above and haven't accumulated much history since), so don't assume "0 scans" on that dashboard means unused without checking the code first.
- `GET /taxonomy` (`routes/taxonomy.py`) returns departments with their nested categories (for building filter UIs) from local `Department`/`Category`/`department_category` tables (`app/models/taxonomy.py`). Categories are many-to-many with departments in SICAR X, not a strict hierarchy. Fetched from SICAR X's `/store/` GraphQL endpoint using an anonymous customer session (`taxonomy_service.fetch_taxonomy_from_sicar`), cached with the same lazy 24h-staleness refresh pattern as `GET /products/{uuid}`.

### Client accounts — a third, distinct auth scheme

`ClientAccount`/`ClientAddress` (`app/models/client.py`, migrations `ad4e748fca2d` + `d8ca048fb64d` for the later `country` column) back `POST /auth/register` and `POST /auth/login` (`routes/auth.py`, `services/client_service.py`) — purely local accounts for the frontend's own login/registration, **unrelated** to either existing SICAR X auth scheme (the admin/B2B token, or `session_service.py`'s per-shopper JWT used for anonymous checkout today). `ClientAccount` has a locally-generated `uuid` (public identifier, separate from the auto-increment `id`), unique `email`, optional `phone`, and `hashed_password` (bcrypt, via `app/core/security.py`'s `hash_password`/`verify_password`). `ClientAddress` is a one-to-many saved address book per account (`client_account_id` FK, `ondelete="CASCADE"`).

Both `/auth/register` and `/auth/login` return `{token, client}` — `token` is a JWT signed with `CLIENT_JWT_SECRET` (`HS256`, `sub` = the client's `uuid`, `exp` per `CLIENT_JWT_EXPIRE_MINUTES`), created by `create_client_token`. `/auth/register` auto-logs-in (returns a token immediately, same shape as `/auth/login`) rather than requiring a separate login call after signup. Duplicate email → `409`; wrong credentials → `401`; both still require the standard `x-api-key` header like every other route (this is frontend-to-middleware auth, not a replacement for it).

`email` is lowercased before every write/read in `client_service.py` (`register_client`/`authenticate_client`) — Postgres's `=` and Pydantic's `EmailStr` are both case-sensitive, so without this, `Juan@x.com`/`juan@x.com` would silently register as two different accounts and logins would fail unpredictably based on casing. `authenticate_client` also always runs `verify_password` — against a fixed dummy bcrypt hash (`_DUMMY_HASH`, computed once at import) when the account doesn't exist — so a nonexistent-email response and a wrong-password response take the same amount of time; skipping the bcrypt call for nonexistent accounts would let an attacker enumerate registered emails by response latency. `POST /auth/login` is rate-limited to 5/minute per IP (`slowapi`, `app/core/rate_limit.py`); `PATCH /auth/me` to 10/minute (covers repeated `current_password` guesses too) — both in-memory, per-process, since there's only one `api` instance today; would need a shared backend (e.g. Redis) if that ever changes.

`GET /auth/me` / `PATCH /auth/me` (`routes/auth.py`) are the first routes protected by this token — `app/core/security.py`'s `get_current_client` dependency decodes the `Authorization` JWT (`Bearer` prefix optional, same tolerant parsing as `session_service.py`), rejects expired/invalid tokens or a deactivated/missing account with `401`, and loads the bare `ClientAccount` (addresses **not** eager-loaded here — see below). `GET` returns the same `ClientPublic` shape as the auth responses — built for a "Mi cuenta" page. `PATCH` (`services/client_service.py`'s `update_client`) supports partial updates to `name`/`phone`; changing `new_password` additionally requires a correct `current_password` in the same request (`401` if it doesn't match) — you cannot change the password on a stolen/stale token alone. Both still require `x-api-key` on top of the client token, same as every route.

`ClientAccount.addresses` deliberately does **not** use `lazy="selectin"` — `get_current_client` runs on every protected route, including `POST`/`PATCH`/`DELETE /auth/me/addresses`, which never touch `client.addresses` at all, so a blanket eager-load would cost every one of those an unneeded extra query. Instead, `Base` mixes in SQLAlchemy's `AsyncAttrs` (`app/core/database.py`), and every code path that actually needs the list (the auth responses, `GET /auth/me`, `GET /auth/me/addresses`) explicitly does `await client.awaitable_attrs.addresses` right before it's needed — this populates the ORM instance's loaded state so a later synchronous access (e.g. during Pydantic's `from_attributes` serialization of `ClientPublic.addresses`) doesn't trigger a lazy load, which would otherwise raise `MissingGreenlet` under the async ORM.

### Address book (`/auth/me/addresses`)

`ClientAddress` is exposed as its own sub-resource (`routes/addresses.py`, `services/address_service.py`) rather than folded into `PATCH /auth/me` — a one-to-many collection needs per-item add/edit/remove, which a single-resource PATCH would force into error-prone client-side diffing (send the whole array, backend guesses what changed). All four routes sit under `/auth/me/addresses`, gated by the same `get_current_client` + `x-api-key` pair as `/auth/me`, and every lookup/update/delete scopes by `client_account_id == current client` — an address `uuid` belonging to a different client resolves as a plain `404`, not `403` (doesn't confirm the row exists at all).

- `GET /auth/me/addresses` — list.
- `POST /auth/me/addresses` — create (`ClientAddressCreate`; `street` required, everything else optional).
- `PATCH /auth/me/addresses/{uuid}` — partial update (`ClientAddressUpdate`, all fields optional).
- `DELETE /auth/me/addresses/{uuid}` — `204` on success.

`ClientAddress.uuid` (migration `56c152338087`) is the public identifier for these routes — added for the same reason every other externally-referenced row in this codebase uses a `uuid` instead of the raw auto-increment `id` (`Product.sicar_uuid`, `ClientAccount.uuid`). "At most one default address per client" is enforced at the DB level with a partial unique index (`ix_client_addresses_one_default`, `UNIQUE (client_account_id) WHERE is_default = true`) rather than only in application code — `address_service.py`'s `_clear_existing_default` unsets any prior default in a separate `UPDATE` *before* the insert/update that sets the new one, so the constraint is never transiently violated; the index is declared directly on the `ClientAddress` model (unlike the `pg_trgm` indexes) so autogenerate won't flag it as a false-positive removal on the next migration.

Deliberately not built yet: no auto-default-on-first-address behavior (the client must explicitly pass `is_default: true`), no email-change flow (changing the login identifier isn't supported by `/auth/me` — would need its own verification flow). `/orders` **does** let a client pick a saved address for `deliveryInfo` now — `deliveryType: "DELIVERYMAN"` references one by `addressUuid` (see "Request flow: placing an order" above); `PICKUP` still doesn't need one. **Planned: Google login.** When that's added, expect `hashed_password` to become nullable (OAuth-only accounts won't have one) and a `sub`/provider marker to distinguish auth methods — not built now since only email/password exists today, but worth knowing before assuming `hashed_password` is always populated.

`ClientAddress` also carries optional `latitude`/`longitude` (migration `bda8078263da`) and `zip_code` gets a lightweight `^\d{5}$` format check in `ClientAddressCreate`/`Update` (`schemas/client.py`). Neither is backed by a server-side integration — **this backend never geocodes or validates postal codes against any external provider**. The frontend's address form resolves zip → state/city/colonias/coordinates itself by calling `envia.com`'s Geocodes API directly (`GET https://geocodes.envia.com/zipcode/{country}/{zipcode}`, `GET https://geocodes.envia.com/locate/{country}/{city}`) — confirmed live to need no API key and to allow CORS from an arbitrary `Origin`, so there's nothing for this API to proxy or protect. `latitude`/`longitude` here just persist whatever coordinates the frontend already resolved, same as any other address field.

Note there's no backend-served county/municipio catalog for a manual "pick a state, then pick a county" flow — envia.com's Geocodes API only does zip/city lookups, not "list all municipios in a state." An earlier pass at this (a Sepomex-based zip proxy plus a bundled INEGI states/municipios JSON file) was built, verified, and then deliberately removed in favor of the envia.com-based approach — if a manual state→county dropdown is needed later, it'll need its own data source, since neither this backend nor envia.com currently serves one.

### Client accounts linked to orders (`/auth/me/orders`)

Orders are now tied to `ClientAccount` — **login is required to place an order**; anonymous checkout no longer exists. This required solving a real conflict: `POST /orders` already used the `Authorization` header for the SICAR X customer-session token (needed to actually talk to SICAR X's cart/order APIs — a real integration requirement, not just an identity check), so there was no header left for the local client-account JWT. Resolution: a **new header, `X-Client-Token`**, carries the client-account JWT on `/orders` and `/cancel` only; `Authorization` keeps its existing meaning there unchanged. Every other `/auth/me*` route is untouched (still reads the client token from `Authorization`, via the existing `get_current_client`). `X-Client-Token` is decoded by the new `get_current_client_header` dependency (`app/core/security.py`) — same JWT/`CLIENT_JWT_SECRET` verification as `get_current_client`, refactored into a shared `_resolve_client_from_token` helper; both are required (not optional) on `/orders`/`/cancel`, so a missing/invalid token is a `401` before any SICAR X call happens. `X-Client-Token` was also added to `main.py`'s CORS `allow_headers` — easy to forget and would otherwise silently fail preflight from a browser frontend.

`Order`/nothing-else (`app/models/order.py`, migrations `45f23f03f9ce` + `9838c19e3da0`) is the **only local record of an order that exists anywhere** — SICAR X itself exposes no endpoint to list a customer's past orders, so this table is now the sole source for order history, not a cache of something queryable elsewhere. `client_account_id` is a required FK (not nullable — matches "login required"), deliberately **without** `ondelete=CASCADE` (unlike `ClientAddress`) since an order is a financial record that shouldn't vanish if account deletion is ever added later (no such feature exists today). `delivery_info`/`items` are snapshotted verbatim from `build_order_payload`'s output at creation time as plain `JSON` (not `JSONB`) — nothing queries inside them, same reasoning as `Product.additional_skus`/`additional_images`. `create_local_order` (`services/order_history_service.py`) is called from `POST /orders` right after `pay_order_in_sicar` succeeds; `OrderResponse` gained a new `orderUuid` field (the local `Order.uuid`) so the frontend can jump straight to `GET /auth/me/orders/{orderUuid}` after checkout without a separate list-and-match step.

`POST /cancel` now also requires `X-Client-Token` and enforces **ownership**: `get_owned_order_by_sicar_id` looks up the local `Order` by `sicar_order_id` scoped to the caller's `client_account_id` *before* calling SICAR X at all, returning `404` (not `403`, same "don't confirm it exists" pattern as `address_service`) if the order doesn't belong to the caller. On successful cancellation the local `order.status` is set to `"CANCELLED"` alongside the existing stock-restore logic.

`GET /auth/me/orders` (list, paginated like `/catalog`) and `GET /auth/me/orders/{order_uuid}` (detail) live in `app/api/routes/client_orders.py`, mirroring `addresses.py`'s router-per-subresource pattern — both gated by the standard `get_current_client` + `x-api-key` (no SICAR session token needed for these, they only read Postgres and optionally make an admin-token SICAR call). List is Postgres-only, matching `/catalog` vs `/products/{uuid}`'s existing split; detail lazily refreshes fulfillment status from SICAR X before responding if the order isn't in a terminal state, matching `GET /products/{uuid}`'s 24h-staleness lazy-refresh pattern (except here there's no time threshold — every non-terminal detail view attempts a refresh, since order status matters in near-real-time and refreshes are cheap single-document lookups, not expensive syncs).

**Fulfillment status is real and richer than PAID/CANCELLED** — confirmed live by capturing traffic from SICAR X's own admin panel (`app.sicarx.com`, the delivery/dispatch board) using `agent-browser`, then reproducing the same query from this codebase's own admin token. The admin panel's dispatch board calls `document-graph/v1/graph-v2` with a `deliveryList` query bucketed by `dispatchStatus`; critically, the **existing** single-document `generatedV2(objectId)` query (already used by `_resolve_document_uuid` for cancellation) also accepts `status`, `dispatchStatus`, and `dispatchHistory` fields — no new endpoint needed, just more fields on a query this codebase already calls. `fetch_document_dispatch_status` (`cancel_service.py`, sibling to `_resolve_document_uuid`) queries exactly that. Confirmed real `dispatchStatus` values, in order: `PENDING_ACCEPTANCE` → `PENDING` → `PREPARING` → `COMPLETE` → `DISPATCHED` (`COMPLETE`/`DISPATCHED` are treated as terminal — `TERMINAL_DISPATCH_STATUSES` in `order_history_service.py` — no more refreshing once reached). `dispatchHistory` is a real audit trail (`date`, `previousStatus`, `currentStatus`, `userName`, `userUuid` per transition) — stored as-is in `Order.dispatch_history` (`JSON`) for a client-facing order-tracking timeline. New REMOTE/PICKUP orders are known to start at `PENDING_ACCEPTANCE` (confirmed live), so `create_local_order` sets that directly instead of spending an extra live SICAR call on the checkout hot path just to confirm it. Note `status` in this GraphQL response (`ACTIVE`/`TO_PAY`/...) is SICAR X's own document-status vocabulary — a **different dimension** from this API's local `Order.status` (`PAID`/`CANCELLED`, this codebase's own payment/cancellation tracking) and intentionally not conflated with it; `refresh_order_status_if_needed` only syncs `dispatch_status`/`dispatch_history`, never overwrites local `status`.

### Payments with Mercado Pago (Checkout Bricks)

`POST /orders` used to pay the SICAR order immediately with a hardcoded `CASH` paymentId via the admin token — no real payment processor was ever involved, it was internal SICAR bookkeeping only. A real Mercado Pago charge now sits between order creation and that bookkeeping step. This repo is backend-only — the Payment/Status Screen Brick UI itself is built in the separate Next.js frontend; this API only exposes the contract it needs (documented in `FRONTEND_INTEGRATION.md`).

Two backend calls are unavoidable, not a design choice: the Brick needs a `preferenceId` (for the Mercado Pago Wallet payment option) and an `amount` *before* it can render — i.e. before the card is tokenized client-side, which only happens in the Brick's own `onSubmit` *after* render.

1. `POST /orders` — validates the cart, creates the SICAR order (`create_order_in_sicar`, reserves stock, unchanged), persists the local `Order` as `status="TO_PAY"`, then calls `payment_service.create_preference` (single aggregate line item — `quantity: 1`, not one per product, since SICAR allows fractional/weight-based quantities that Mercado Pago's preference items don't support well). Returns `id`/`orderUuid` (as before) plus `preferenceId` and `amount`.
2. Frontend renders the Payment Brick with `initialization.{amount, preferenceId}`.
   - **Card / OXXO submit** (the Brick's `onSubmit` fires client-side, synchronous MP response) → frontend calls **`POST /orders/{order_id}/pay`** with the Brick's `formData` (`token`, `paymentMethodId`, `issuerId`, `installments`, `payer`) — mirrors `/cancel`'s ownership pattern (`get_owned_order_by_sicar_id`, requires `X-Client-Token`, `404` if not owned). `409` if the order isn't still `TO_PAY` (already paid/cancelled — blocks double-charge). `payment_service.create_payment` charges Mercado Pago's `POST /v1/payments`; **`transaction_amount` always comes from `order.total` already in Postgres, never from the request body** — same "don't trust the client's price" principle `build_order_payload` already applies elsewhere. `order.uuid` doubles as the `X-Idempotency-Key` so a retried submit dedupes on Mercado Pago's side instead of double-charging.
   - **Wallet button** → user is redirected straight to Mercado Pago's own site; per Mercado Pago's docs this **never calls our backend at submit time at all** — the only way this API learns the outcome is the webhook.

Both paths funnel into the same `order_history_service.finalize_order_payment(db, order, mp_payment)` — the one place that maps a Mercado Pago payment status onto local order state (never duplicate this branching):
- `approved` → runs `pay_order_in_sicar` (unchanged, still the SICAR-ledger bookkeeping step) → local `status = "PAID"`.
- `pending`/`in_process` (OXXO ticket awaiting cash payment, card under review, wallet not yet completed) → **does not** call `pay_order_in_sicar` → local `status` stays `"TO_PAY"`. `mp_ticket_url` (from `transaction_details.external_resource_url`) is stored so an OXXO barcode link can be surfaced later.
- `rejected`/`cancelled` → calls `process_order_cancellation` (existing, from `cancel_service.py`) to release the reserved stock and cancel the SICAR document, built from `order.items` already on the row (the caller doesn't need to resend the cart, unlike the manual `/orders/{id}/cancel` endpoint) → local `status = "CANCELLED"`.

`Order.status` vocabulary is now `{TO_PAY, PAID, CANCELLED}` (was `{PAID, CANCELLED}`) — `TO_PAY` is the default and mirrors SICAR X's own native document-status vocabulary (`ACTIVE`/`TO_PAY`/... via `generatedV2`, already referenced in `cancel_service.py`).

**`POST /payments/webhook`** (`app/api/routes/payments.py`) is the only router in this codebase **without** `Depends(validate_api_key)` — Mercado Pago's servers can't send our static `x-api-key`. Authenticity instead comes entirely from `payment_service.verify_webhook_signature` (HMAC-SHA256 over a manifest string keyed by `MP_WEBHOOK_SECRET`, checked against the `x-signature`/`x-request-id` headers) — rejects before doing anything else if it fails. **Caveat**: the exact manifest format is no longer published verbatim on Mercado Pago's current docs (they push the official SDK instead, which is synchronous and therefore not used here — every other external integration in this codebase, SICAR X included, uses plain `httpx` for the same non-blocking-event-loop reason). The implemented format (`id:{data.id};request-id:{x-request-id};ts:{ts};`, lowercased `data.id`) is corroborated by multiple third-party/community sources but **has not yet been verified against a real Mercado Pago notification** — verify with the dashboard's webhook simulator before trusting it in production, and fix `verify_webhook_signature` if a live delivery doesn't match. On success, the webhook re-fetches the payment via `payment_service.get_payment` (never trusts the notification body itself for the actual status — Mercado Pago's own recommended practice) and calls the same `finalize_order_payment`. Always responds `200` even on a no-op (unknown order, already-terminal status) — Mercado Pago retries aggressively on any non-2xx.

`POST /orders/{order_id}/cancel` also got a payment-cleanup step: before the existing SICAR-side cancellation, if the order has an `mp_payment_id`, it refunds (`payment_service.refund_payment`) an already-`approved` payment or cancels (`payment_service.cancel_payment`) a still-`pending`/`in_process` one — money-side resolves before the SICAR bookkeeping side, same ordering principle as `pay_order_in_sicar` already being "apply the real-world fact to SICAR's ledger."

### Auth on this API's own endpoints

Every route depends on `validate_api_key` (`app/core/security.py`), which checks a static `x-api-key` header against `settings.X_API_KEY`. This is separate from both SICAR X token schemes above — it authenticates the frontend to this middleware, not this middleware to SICAR X.

## Logging

- The FastAPI app logs to `app.log` (configured in `main.py`).
- The worker logs separately to `sync.log` via a `RotatingFileHandler` (10MB × 5 backups) configured in `app/worker/sync_task.py` — the two processes do not share a logger config.
- Log messages and docstrings throughout are in Spanish; keep new code consistent with that convention.
